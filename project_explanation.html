Project1 — Technical Breakdown

A multi-module Android application that fetches achievements from a remote API and displays them using a clean MVVM architecture.

Kotlin 2.2.0 MVVM Retrofit 2 Coroutines LiveData Data Binding Glide 5 Multi-Module

Project Structure — Three Modules

The project is split into three Gradle modules. Each module has a single responsibility. This separation means the UI layer cannot directly talk to the network — it must go through the data layer, which enforces discipline and makes each part independently testable.

:app

The Android Application module. Depends on both :data and :core. Knows nothing about Retrofit or the network directly.

:data

An Android Library module. Owns all network communication, models, and the Repository. Has no knowledge of any UI component.

:core

An Android Library module for shared utilities used across modules. Currently holds the Glide image-loading binding adapter.

Architecture Pattern — MVVM + Repository

The app follows MVVM (Model–View–ViewModel) combined with the Repository pattern. The idea is that each layer only knows about the layer directly below it. The UI never talks to the network. The ViewModel never builds HTTP requests.

API Connection — Retrofit

Retrofit is a type-safe HTTP client for Android. You describe your API as a Kotlin interface with annotated functions, and Retrofit generates the code that actually makes the network call.

Step 1 — RetrofitBuilder (the HTTP engine)

RetrofitBuilder is a Kotlin object (a singleton — only one instance ever exists). It configures Retrofit with two things: where to send requests (the base URL) and how to parse the JSON response (Gson). 

object RetrofitBuilder {
    private const val baseUrl = "https://786b905e-735c-4be6-adfb-949d5dadee32.mock.pstmn.io/"

    fun getInstance(): Retrofit {
        return Retrofit.Builder()
            .baseUrl(baseUrl)              // all requests go to this root URL
            .addConverterFactory(GsonConverterFactory.create()) // JSON → Kotlin data classes
            .build()
    }
}
Why a Kotlin object? An object in Kotlin is a singleton by language definition. There is no constructor — the first time you reference it, it is created, and every subsequent reference returns the same instance. This means Retrofit is only configured once for the entire app lifetime.

Step 2 — ProjectApi (the endpoint declaration)

This is just an interface. You write the endpoint as a function — Retrofit reads the annotations and generates the full HTTP implementation for you at runtime. No manual URL building, no manual parsing. 

interface ProjectApi {

    @GET("/achievements")
    suspend fun getAchievements(): WrappedAchievementsResponseModel
}
What does suspend mean here? It means this function can only be called from a coroutine. Retrofit detects the suspend keyword and automatically runs the network request off the main thread, then resumes the coroutine with the result when the response arrives — no callbacks, no threads to manage manually.

Step 3 — JSON → Data Classes (Gson)

The server returns JSON. Gson maps each JSON field to a property in a Kotlin data class automatically, matching by field name. The @SerializedName annotation handles cases where the JSON key differs from the Kotlin property name. 

// What the server returns (simplified JSON):
// {
//   "data": [
//     { "id": 1, "title": "Personal Records", "label": "4 of 6", "records": [...] }
//   ]
// }

// The outer wrapper — matches the top-level JSON object
data class WrappedAchievementsResponseModel(
    val data: List<AchievementsResponseModel>
)

// Each achievement in the "data" array
data class AchievementsResponseModel(
    @SerializedName("id")       // JSON key is "id", Kotlin name is idAchievements
    val idAchievements: Int,
    val title: String,
    val label: String,
    val records: List<RecordsModel>
)

// Each medal record nested inside an achievement
data class RecordsModel(
    val id: Int,
    val title: String,
    val label: String,
    val active: Boolean,
    val image: String   // URL string, loaded later by Glide
)

Repository — The Single Source of Truth

The Repository is a plain class that sits between the ViewModel and the data sources (network, database, cache, etc.). Its job is to hide where data comes from. The ViewModel just asks "give me the achievements" — it does not care if they come from the network, a local cache, or a database. 

class Repository {

    suspend fun getAchievements(): List<AchievementsResponseModel> {
        val achievementsApi = RetrofitBuilder.getInstance().create(ProjectApi::class.java)
        return achievementsApi.getAchievements().data
    }
}

Three things happen on that one line inside the function:

1. Build the API

RetrofitBuilder.getInstance().create(ProjectApi::class.java) — Retrofit reads the ProjectApi interface and generates a real implementation of it at runtime using reflection.

2. Make the HTTP call

.getAchievements() — triggers the actual GET /achievements network request. Because it's suspend, the coroutine waits here without blocking any thread.

3. Unwrap the response

.data — the server wraps the list inside a { "data": [...] } object. The Repository strips that wrapper and returns only the list, so the ViewModel never has to know about the envelope.

Why have a Repository at all? Without it, the ViewModel would directly call Retrofit. That makes testing impossible (you can't swap out a real network call for a fake one) and mixes responsibilities. With a Repository, you could later add a local Room database as a cache — the ViewModel would never need to change, because the Repository decides which source to use.

Coroutines — Async Without the Pain

Android's main thread drives the UI. If you do a network request on it, the app freezes and Android will kill it ("Application Not Responding"). You need to do the request on a background thread and then bring the result back to the main thread to update the UI.

Traditionally this required callbacks, RxJava streams, or manual thread management. Kotlin Coroutines let you write that async code as if it were sequential — no nested callbacks, no explicit thread switching. 

class MainViewModel : ViewModel() {

    private val repository = Repository()

    val achievementsSuccessLiveData: MutableLiveData<List<AchievementsResponseModel>> = MutableLiveData()
    val achievementsErrorLiveData: MutableLiveData<Exception> = MutableLiveData()

    init {
        getAchievements()  // called automatically when the ViewModel is first created
    }

    fun getAchievements() {
        viewModelScope.launch {           // 1. start a coroutine
            try {
                val response = repository.getAchievements()  // 2. suspend — waits for network
                achievementsSuccessLiveData.postValue(response) // 3. deliver result
            } catch (e: Exception) {
                achievementsErrorLiveData.postValue(e)          // 4. deliver error
            }
        }
    }
}

viewModelScope

A coroutine scope tied to the ViewModel's lifecycle. When the ViewModel is destroyed (e.g. app closed), all coroutines launched in this scope are automatically cancelled — no memory leaks.

launch { }

Starts a new coroutine. Code inside runs asynchronously. The launch call itself returns immediately — it doesn't wait for the block to finish.

suspend

A function marked suspend can pause ("suspend") without blocking the thread. The thread is free to do other work while waiting. When the result is ready, the coroutine resumes from exactly where it left off.

try / catch

Error handling in coroutines works like normal Kotlin. If the network fails, the catch block runs and the error LiveData is updated — no separate error callbacks needed.

Coroutines vs. Threads: A thread is an OS resource — you can have hundreds at most before performance degrades. A coroutine is lightweight — you can have tens of thousands. Coroutines also know how to switch between threads automatically (Retrofit uses Dispatchers.IO internally). This project never manually creates a thread or manages an executor.
Note on Flows: This project uses LiveData as the reactive stream rather than Kotlin Flows. LiveData is lifecycle-aware — it only delivers updates when the Activity/Fragment is in an active state (started/resumed), preventing crashes from updating a destroyed UI. Flows are more powerful and the modern recommendation, but LiveData is simpler and sufficient here.

ViewModel — Surviving Rotation

Every time you rotate your phone, Android destroys and recreates the Activity. Without a ViewModel, that would cancel your network request and lose all your data. The ViewModel lives independently of the Activity's lifecycle — it survives rotation and is only destroyed when the user actually leaves the screen (presses back).

ViewModelProvider(this)[MainViewModel::class.java] in MainActivity is the key call. If a ViewModel already exists for this Activity (e.g. after a rotation), the existing one is returned. If not, a new one is created and init { getAchievements() } runs. This means the network request only fires once, not every time the screen rotates.

LiveData — The Two Channels

The ViewModel exposes two MutableLiveData streams — one for success, one for errors. The Activity observes both and reacts accordingly. 

// In MainViewModel — publish data
achievementsSuccessLiveData.postValue(response)   // thread-safe, can be called from coroutine
achievementsErrorLiveData.postValue(e)

// In MainActivity — observe data
viewModel.achievementsSuccessLiveData.observe(this) { response ->
    binding.recyclerViewAchievement.adapter = AchievementAdapter(response)
}
viewModel.achievementsErrorLiveData.observe(this) { e ->
    Toast.makeText(this, "error=${e.localizedMessage}", Toast.LENGTH_LONG).show()
}
postValue vs setValue: postValue() is thread-safe and can be called from a background thread (the coroutine). setValue() must be called from the main thread. Since the coroutine runs off-thread, postValue() is used here.

Data Binding — Model → UI Automatically

Normally you find views with findViewById() or view binding, then manually set each property: textView.text = model.title. Data Binding lets you write those assignments directly in the XML layout, so the framework generates the glue code for you.

How a Data Binding Layout Works

Any layout wrapped in a <layout> tag becomes a data binding layout. Inside it, you declare a <data> block with variables, and use @{ } expressions to bind them directly to view properties. 

<!-- row_achievements.xml — the outer achievement row -->
<layout>
    <data>
        <variable
            name="model"
            type="com.training.ptojectone.data.model.AchievementsResponseModel" />
    </data>

    <ConstraintLayout ...>
        <AppCompatTextView
            android:text="@{model.title}"   <!-- bound directly -->
            ... />
        <AppCompatTextView
            android:text="@{model.label}"   <!-- bound directly -->
            ... />
    </ConstraintLayout>
</layout>
The Android build system reads every <layout> XML and generates a Kotlin class for it. row_achievements.xml → RowAchievementsBinding. row_medal.xml → RowMedalBinding. activity_main.xml → ActivityMainBinding (view binding — no <layout> tag needed).

Using Binding in Adapters

Instead of inflating a plain View and calling findViewById, the adapters inflate the Binding class. Then setting data is one line: binding.model = model — that single assignment triggers all the @{ } expressions in the XML to update. 

class AchievementAdapter(private val list: List<AchievementsResponseModel>) :
    RecyclerView.Adapter<AchievementAdapter.MedalViewHolder>() {

    override fun onCreateViewHolder(parent: ViewGroup, viewType: Int): MedalViewHolder {
        // Inflate the binding class — not just a View
        val binding = RowAchievementsBinding.inflate(
            LayoutInflater.from(parent.context), parent, false
        )
        return MedalViewHolder(binding)
    }

    inner class MedalViewHolder(private val binding: RowAchievementsBinding) :
        RecyclerView.ViewHolder(binding.root) {

        fun bind(model: AchievementsResponseModel) {
            binding.model = model   // one line — updates ALL bound views in the XML
            binding.recyclerViewMedal.adapter = RecordsAdapter(model.records) // nested RecyclerView
        }
    }
}

Custom Binding Adapter — Loading Images with Glide

Data binding knows how to set text, visibility, colors, etc. out of the box. But loading an image from a URL is not a built-in operation — you need a custom Binding Adapter. A Binding Adapter is a function annotated with @BindingAdapter that teaches the data binding framework how to handle a custom XML attribute. 

// core/BindingAdapters.kt
@BindingAdapter("imageUrl")                  // matches app:imageUrl in XML
fun ImageView.imageUrl(url: String?) {
    Glide.with(this.context)
        .load(url)                           // download image from URL
        .into(this)                          // place it into the ImageView
}
<!-- row_medal.xml — used automatically when app:imageUrl is set -->
<AppCompatImageView
    app:imageUrl="@{model.image}"   <!-- triggers the @BindingAdapter above -->
    android:layout_width="110dp"
    android:layout_height="110dp"
    ... />
Glide handles the entire image pipeline: downloads the image on a background thread, caches it in memory and on disk so it's not re-downloaded on scroll, decodes it to the correct size, and places it in the ImageView on the main thread — all triggered by that single app:imageUrl attribute.

Full Data Flow — End to End

Here is the complete journey data takes from the server to the screen:

① App Launches → ViewModel Created

MainActivity.onCreate() calls ViewModelProvider(this)[MainViewModel::class.java]. A new MainViewModel is created. Its init { } block immediately calls getAchievements().

② Coroutine Starts

viewModelScope.launch { } starts a coroutine on the main thread dispatcher. Execution enters the coroutine body and hits repository.getAchievements(). Because this is a suspend function, the coroutine suspends — the main thread is released and is free to render frames while waiting.

③ HTTP Request

Retrofit executes the GET /achievements request on a background I/O thread. The raw JSON response comes back from the Postman mock server. Gson automatically deserializes it into WrappedAchievementsResponseModel.

④ Repository Unwraps

The Repository calls .data on the wrapped model, stripping the outer envelope. It returns a plain List<AchievementsResponseModel> to the ViewModel. The coroutine resumes.

⑤ LiveData Updated

achievementsSuccessLiveData.postValue(response) posts the list to LiveData. LiveData checks whether the Activity is in an active state (started/resumed). If it is, it delivers the value to the observer.

⑥ RecyclerView Populated

The observer in MainActivity receives the list and sets an AchievementAdapter on the RecyclerView. The adapter creates RowAchievementsBinding instances for each row. binding.model = model populates the title and label via data binding. A nested RecordsAdapter is set on the inner RecyclerView inside each achievement row.

⑦ Images Load Asynchronously

Each medal row has app:imageUrl="@{model.image}" in its layout. The custom binding adapter is called with the URL string. Glide fetches the image on a background thread, caches it, and places it in the AppCompatImageView when ready — completely independent of the main data flow.
